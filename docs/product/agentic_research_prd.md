# PaperWiki Agentic Research PRD

版本：v0.1

状态：实施前产品基线

适用范围：课程大作业重构与答辩演示

## 1. 产品定义

### 1.1 一句话定位

PaperWiki 是一个由 AI Agent 驱动的科研论文工作台：用户用自然语言提出研究任务，系统自主检索、抓取、解析、阅读和比较论文，以可视化 Workflow 展示执行过程，并把可信结果沉淀为可检索、可引用、可继续追问的个人论文知识库。

### 1.2 核心创新点

传统论文工具要求用户依次点击“搜索、导入、解析、阅读、总结、收藏、写报告”。PaperWiki 把这些能力包装成 Agent 可调用工具。用户只需表达目标，例如：

> 帮我调研 2023 年以来 RAG 检索优化相关的论文，重点比较召回策略、评测数据集和主要局限，最后生成一份带出处的调研报告。

系统会先判断任务是否足够明确，然后自动执行完整流程。用户始终能看到：

- Agent 当前在做什么；
- 使用了哪个工具；
- 找到了哪些论文；
- 为什么筛选或排除；
- 哪些结论来自哪些正文证据；
- 哪一步失败、重试或需要用户决定；
- 最终产物保存在哪里。

### 1.3 不是哪些产品

- 不是只有聊天外观、内部仍是单轮问答的 Chatbot。
- 不是完全无人监督的 AutoResearch；课程版本不自主提出研究 Idea、判断学术创新性或复现实验。
- 不是论文搜索结果页的视觉包装；核心价值是可执行 Workflow 和可信产物。
- 不展示模型隐藏思维过程。界面只展示可审计的计划摘要、工具调用、状态、结果和证据。

## 2. 目标用户与核心任务

### 2.1 目标用户

| 用户 | 主要痛点 | PaperWiki 提供的价值 |
| --- | --- | --- |
| 课程学习者 | 不知道从哪些论文入门，搜索与阅读割裂 | 用自然语言启动主题调研，获得代表论文和学习路径 |
| 论文初读者 | PDF 长、术语多、方法与实验难定位 | 结构化解析、单篇问答、证据定位与概要 |
| 开题/综述准备者 | 搜索、筛选、对比、引用整理耗时 | 自动完成候选收集、对比矩阵与带出处报告 |
| 持续关注某方向的用户 | 新论文多、收藏后难整理 | 主题资料库、后续订阅与增量调研 |

### 2.2 Jobs to be Done

1. 当我进入一个不熟悉的研究主题时，我希望系统帮我找到入口、代表论文和关键概念。
2. 当我准备课程报告或开题时，我希望系统自动收集并比较论文，同时保留出处。
3. 当我阅读一篇论文时，我希望直接围绕正文提问，并跳到相关内容。
4. 当我积累论文时，我希望系统帮我整理成可继续检索和复用的知识资产。
5. 当 Agent 长时间工作时，我希望知道它做到哪一步、是否卡住，以及结果是否可信。

## 3. 产品原则

1. **自然语言是启动方式，不是唯一界面。** Chat、论文库、资料库三个主入口并列存在。
2. **默认自动，关键时刻确认。** 普通检索、抓取、解析、真实模型调用、生成草稿和私有保存自动执行。
3. **过程可见但不过载。** 默认展示一句话进度；用户展开后查看工具参数、结果、证据与错误。
4. **结论必须可追溯。** 报告中的事实性结论必须绑定已打开的论文证据。
5. **失败可恢复。** Run 支持暂停、继续、重试失败步骤和从 checkpoint 恢复。
6. **产物可沉淀。** 调研结果不是一次性消息，而是可打开、保存、继续追问的 Artifact。
7. **课程要求可验证。** 需求覆盖必须有自动测试、性能数据或人工验收证据，不以界面文案代替完成状态。

## 4. 信息架构

三个主入口并列：

| 主入口 | 用户心智 | 核心内容 |
| --- | --- | --- |
| Chat | “告诉系统我要做什么” | 普通问答、Agent 调研、任务执行、结果交付 |
| 论文库 | “查看系统里有什么论文” | 搜索、筛选、同步、上传、批量操作、进入阅读器 |
| 我的资料库 | “管理我留下的研究资产” | 文件夹、收藏论文、调研项目、报告、历史任务 |

二级对象通过深链接访问，不增加第四个常驻主入口：

- `/runs/:runId`：调研任务详情；
- `/papers/:paperId`：单篇论文阅读工作台；
- `/reports/:reportId`：调研报告；
- `/collections/:collectionId`：调研论文集合；
- `/graphs/:graphId`：主题脉络或概念图谱。

全局顶部栏提供“任务中心”按钮，显示运行中和等待用户的任务数量。它是抽屉/面板，不是新的主导航。

## 5. 核心场景

### 5.1 场景 A：主题调研（旗舰 Workflow）

用户输入：

> 帮我调研 LLM Agent 的长期记忆相关论文，优先 2024 年以后的工作，比较记忆写入、检索和评测方式。

目标流程：

1. 解析研究目标与约束。
2. 如果存在会显著改变结果的歧义，只问一个关键问题。
3. 生成可执行计划和预算。
4. 检索 arXiv 与已有论文库，合并去重。
5. 根据相关性、时间、来源和可获取性初筛。
6. 抓取并导入候选论文。
7. 解析 PDF/HTML，失败时重试或降级为可说明的状态。
8. 抽取研究问题、方法、数据集、实验结论、局限和核心概念。
9. 对比论文并识别主题分组、代表论文和研究空白。
10. 生成带证据引用的调研报告。
11. 校验引用完整性与关键结论支持度。
12. 将论文集合、报告和图谱默认保存到用户私有资料库。

最终交付：

- 一份调研报告；
- 一组可筛选的候选/入选论文；
- 一张方法对比矩阵；
- 一条可展开的 Workflow 执行记录；
- 可选主题脉络图和概念图谱；
- “继续研究”建议，例如缩小范围、补充另一类论文或订阅主题。

### 5.2 场景 B：单篇精读

用户可从 Chat 输入论文链接/标题，也可从论文库进入详情页。系统自动完成导入、解析、生成概要，并在阅读工作台中打开 PDF、结构化内容和论文 Chat。

### 5.3 场景 C：论文对比

用户指定两篇或多篇论文，Agent 检查正文可用性，抽取统一字段，生成对比矩阵，并允许从单元格跳转到证据。

### 5.4 场景 D：普通问答

不需要工具的知识问题继续走普通 Chat。系统先分类意图，不应把每条消息都强制升级为长 Workflow。用户可以手动点击“转为调研任务”。

## 6. Agent 自主执行与确认策略

### 6.1 默认自动执行

以下动作默认无需确认：

- 搜索公开论文元数据；
- 抓取公开论文 PDF/HTML；
- 去重、筛选和排序；
- 调用已配置的真实 LLM；
- 解析论文、生成结构化知识与私有报告草稿；
- 将任务产物保存到当前用户私有资料库；
- 在预算内自动重试网络、解析和模型瞬态错误。

### 6.2 必须确认

只有以下情况进入 `waiting_input`：

- 研究主题存在会显著改变检索集合的关键歧义；
- 用户要求的时间、来源、论文数量或输出格式互相冲突；
- 预计调用成本或候选规模超过预设预算；
- 需要公开发布、分享、覆盖现有报告或删除数据；
- 需要扩大到用户未授权的数据源或工具；
- 多种研究口径都合理且系统无法安全择优；
- 连续失败后需要用户决定跳过、替换来源或终止。

### 6.3 确认交互要求

- 每次只问一个最关键问题。
- 提供推荐选项及影响说明。
- 用户回答后从原步骤继续，不重新启动整条任务。
- 等待期间允许用户离开页面，任务中心持续显示“需要你确认”。

## 7. 多 Agent 与工具设计

### 7.1 角色

| Agent | 职责 | 主要输入 | 主要输出 |
| --- | --- | --- | --- |
| Research Coordinator | 理解意图、规划、编排、控制预算和收尾 | 用户目标、权限、Run 状态 | Plan、步骤调度、最终交付 |
| Search Agent | 构造检索式、跨来源搜索、去重与召回 | 主题、筛选条件 | 候选论文集合 |
| Screening Agent | 根据统一标准打分、入选/排除并说明原因 | 候选论文 | 入选列表、排除理由 |
| Reader Agent | 获取正文、定位章节与证据 | PaperDocument | 阅读卡片、证据候选 |
| Extraction Agent | 抽取方法、数据集、结论、局限和概念 | 正文与 schema | 结构化 Paper Brief |
| Synthesis Agent | 聚类、比较、生成研究脉络和报告 | 多篇 Paper Brief | 矩阵、趋势、报告草稿 |
| Citation Validator | 校验结论、引用和证据闭环 | 报告、Evidence Registry | 校验结果、修订建议 |
| Librarian Agent | 整理文件夹、保存私有产物和建立关联 | Artifact 集合 | 资料库结构、保存结果 |

首版可以按步骤顺序调用不同 Agent，不要求所有角色并行。课程答辩中的“多 Agent 协同”应通过明确的角色、输入输出 schema、交接记录和 UI 状态体现，而不是只在 trace 中列出几个名字。

### 7.2 Tool Registry

首版工具建议：

- `search_metadata`：搜索 arXiv、现有库及后续来源；
- `ingest_papers`：导入并去重；
- `fetch_document`：获取 PDF/HTML；
- `parse_document`：Docling 解析；
- `search_text`：正文检索；
- `open_evidence`：打开并登记可引用证据；
- `extract_paper_brief`：生成统一结构；
- `compare_papers`：构造对比矩阵；
- `build_topic_graph`：构造主题/概念关系；
- `write_report`：生成报告草稿；
- `validate_citations`：验证引用；
- `save_artifacts`：保存到用户私有资料库。

工具必须声明输入 schema、输出 schema、权限等级、是否幂等、超时、重试策略和可展示摘要。

## 8. 功能需求

### 8.1 Chat 与任务

- 支持普通对话和 Agent 调研两类运行模式，由系统自动路由，用户可手动切换。
- 每次 Agent 调研创建持久化 Research Run。
- Run 支持后台运行、暂停、继续、停止、重试失败步骤和复制任务。
- 对话消息可引用 Run 和 Artifact；Run 不依赖页面一直打开。
- 用户刷新或重新登录后能恢复任务状态。

### 8.2 Workflow 可视化

- 展示总进度、当前步骤、运行时间、论文计数、工具调用数和错误/等待状态。
- 步骤支持折叠和展开；展开后显示 Agent、工具、输入摘要、输出摘要、证据和错误。
- 支持状态分组：需要确认、正在执行、已完成、失败/已停止。
- 支持查看完整事件时间线，但默认不展示冗长原始日志。

### 8.3 论文与知识

- 保留论文库搜索、导入、上传和详情工作台。
- Agent 自动导入的论文带 `run_id`、来源和筛选理由。
- Paper Brief 使用统一字段，支持跨论文对比。
- 报告引用必须能跳转到论文、章节/片段和 PDF/解析文本位置。

### 8.4 资料库与产物

- 资料库同时容纳论文、论文集合、调研报告和图谱。
- 每次调研默认生成一个私有项目目录。
- 报告支持继续追问、复制 Markdown、导出和重新运行局部步骤。
- 删除、公开和覆盖已有产物需要确认。

## 10. 非功能需求

### 10.1 性能

- 创建 Research Run 的接口在 500ms 内返回 Run ID，不等待 LLM 完成。
- 论文列表、资料库、任务列表等非 LLM API 在 100 并发下 p95 小于 3 秒。
- SSE 事件在服务端状态变化后 1 秒内到达前端。
- 大型 Workflow 不阻塞 FastAPI 请求线程；解析和模型任务进入后台执行层。

### 10.2 准确性

- 检索召回、入选准确、字段抽取和引用支持度分开评测，不能用一个模糊“准确率”替代。
- 课程声明的 >90% 必须绑定一个固定 gold set 和明确指标。
- 报告中带事实判断的段落至少包含一个已登记 Evidence。
- Citation Validator 失败时不得把报告标记为“已验证”。

### 10.3 安全与权限

- API Key 仅通过环境变量或部署密钥注入。
- 私有上传、私有报告和 Run 仅归当前用户访问。
- 论文正文视为不可信输入，不能改变系统权限或工具策略。
- 工具参数、错误和 UI 日志不得泄漏密钥、Cookie、原始 Authorization 或未经清洗的 provider body。

### 10.4 可恢复性

- 每个步骤完成后保存 checkpoint。
- 重试必须幂等，不能重复创建论文、报告或资料库条目。
- 后端重启后，未完成 Run 可以恢复为可继续或明确失败，不允许长期假装运行中。

## 11. 成功指标

### 11.1 旗舰体验指标

- 用户从一条自然语言指令启动完整调研。
- 无关键歧义时不出现多余确认。
- 任务过程中至少展示 6 个可辨识步骤和实时状态。
- 最终报告至少引用 5 篇论文，并能从引用打开证据。
- 任务结果自动保存到私有资料库。
- 用户能暂停/恢复，能对失败步骤单独重试。

### 11.2 产品质量指标

- 真实调研任务完成率 ≥ 90%。
- 有效引用支持率 ≥ 90%。
- 失败步骤可诊断率 100%，不得只显示“发生错误”。
- 关键操作在桌面与 390px 移动宽度均可完成。
- 页面无未处理的横向溢出，键盘焦点可见。

## 12. 首版范围与非目标

### 12.1 首版必须完成

- Chat 首页接入 Research Run。
- 持久化 Workflow、事件、决策和产物。
- 主题调研完整链路。
- 可视化 Workflow 面板。
- 报告、论文集合和引用证据。
- 私有资料库沉淀。
- 真实模型、真实 arXiv、真实浏览器演示。

### 12.2 可后置

- 多来源会议网站的统一高级检索。
- 每日订阅推送。
- 多用户团队空间与共享协作。
- 完整管理员审核系统。
- 真正并行的多 Agent 执行。
- 复杂图表、公式和图片多模态理解。

### 12.3 明确不做

- 自主提出研究创新点并声称新颖。
- 自动复现论文代码和实验。
- 无证据的“研究趋势”断言。
- 未经确认公开用户资料或删除数据。

## 13. 产品验收场景

以“2023 年以来 RAG 检索优化论文调研”为标准验收任务：

1. 用户在 Chat 输入一句话要求。
2. 系统自动建立 Run，并在 1 秒内展示 Workflow。
3. 系统依次显示计划、搜索、筛选、导入、解析、阅读、综合、校验和保存。
4. 用户可展开任一步查看工具与结果摘要。
5. 若搜索范围足够明确，全程不要求确认。
6. 最终报告包含代表论文、方法分类、对比矩阵、局限、趋势和引用。
7. 每个引用可跳转到论文证据。
8. 结果出现在“我的资料库”对应调研项目中。
9. 用户刷新页面后 Run 和报告仍存在。
10. 用户点击“基于此报告继续研究”可开启关联新 Run。

## 14. Iter15 项目化研究脉络决策

- 研究项目属于“我的资料库”，不新增第四个顶级导航。项目只能组织当前 Session 已可访问的 Run、论文和固定 Report version，不构成新的授权边界。
- 项目分析输出是版本化主题簇、时间线和关系图。事实性节点/边必须绑定当前有效 Citation、PaperBrief Evidence 或已验证论文元数据；模型自由文本不是数据库关系真相。
- publication/precedes 只表达已验证日期和时间排序；“改进、反驳、影响、转折”等语义必须有 Citation。当前无可靠 bibliography 数据，v1 不依标题相似度生成 `cites` 边。
- 项目输入、Citation/Evidence、ACL 或 source hash 变化后，旧分析立即投影 stale/inaccessible；受影响事实文本被隐藏，历史版本只保留安全审计信息。
- 资料不足时在付费调用前进入 Coverage Decision，允许添加资料、移除失效项、生成有限脉络、减少维度、仅确定性时间线、返回编辑或停止。有限模式必须明示 coverage 和缺失项。

## 15. Iter16 质量评测决策

- Citation entailment 评测与当前结构型 Citation Validator 分离：前者是离线开发者验收工具，后者继续只负责 locator/hash/ACL/source 关系。评测结果不回写生产数据库，也不放宽报告 exact-statement 白名单。
- 固定数据集使用 2023 年以来 5 篇公开 RAG 论文和 60 个 adjudicated exact-statement 案例；三标签各 20 条。语义 coverage 只统计报告/矩阵事实、Cluster 摘要/区分特征、非 publication 时间线语义和 Graph 语义边；publication 与确定性关系单列。
- `prediction_file` 用于确定性复算指标，但即使达到阈值也显示“未验证”；只有本次经授权的真实 `llm_judge` 且 macro-F1、supported precision、coverage 同时达到阈值才显示“已达标”。
- 真实 judge 每案例只允许一次 provider 请求；缺少配置、schema/case ID 不一致、provider 失败或敏感输出均 fail closed。报告只保留模型标识、版本/hash 和脱敏配置。
- 答辩连续验收路径固定为 Chat → topic Run → 固定报告 → Citation Evidence → 论文 Chunk 定位 → 研究项目 → Graph Evidence，并在 1440/1024/390 下验证键盘、焦点、reduced-motion 和无溢出。
